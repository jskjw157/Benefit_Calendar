# 혜택 캘린더 (BenefitCal) MVP 정의 및 실행 계획

## 1) MVP 정의

### 1.1 MVP 목표
- 신규 사용자가 **3분 이내**에 개인화된 혜택 리스트를 확인하도록 한다.
- 검색/필터/캘린더로 **혜택 탐색**과 **마감 관리**를 가능하게 한다.
- 북마크 및 진행 상태 변경으로 **신청 준비 흐름**을 지원한다.

### 1.2 대상 사용자 (MVP)
- **청년 지수 (25세)**: 청년지원금/수당 중심.
- **자영업자 민수 (38세)**: 소상공인 지원금 중심.

### 1.3 MVP 범위 (In Scope)
**핵심 사용자 여정**
1. **온보딩/프로필 설정**: 나이, 지역, 직업 상태 입력 → 추천 혜택 제공
2. **혜택 발견/탐색**: 검색, 카테고리/지역 필터, 정렬
3. **혜택 상세 확인 및 신청 준비**: 상세 정보, 제출 서류, 외부 신청 링크
4. **알림/재방문**: 마감일 임박 알림 설정 (이메일/앱 알림 중 1개)

**필수 화면**
- 홈/대시보드
  - 개인화 추천, 마감 임박, 신규 혜택, 통계 위젯(매칭 수/마감 임박/신청 완료)
- 혜택 탐색
  - 검색 바, 필터(카테고리/지역/신청 가능), 정렬, 결과 수
- 혜택 상세
  - 지원 기관, 금액/신청 기간, 자격 요건, 제출 서류, 신청 링크
- 캘린더
  - 월별 뷰, 마감 임박 강조, 날짜 클릭 시 해당 혜택 리스트
- 내 혜택
  - 북마크/준비중/신청완료/수령완료 탭, 상태 변경
- 설정
  - 프로필 편집, 알림 설정

**핵심 데이터**
- **User**: id, 이메일, 나이, 지역, 직업 상태(자영업 여부), 알림 수신 설정
- **Benefit**: id, 제목, 기관, 카테고리, 지역, 지원 금액, 신청 기간, 마감일, 신청 링크
- **UserBenefit**: user_id, benefit_id, 상태(북마크/준비중/신청완료/수령완료), 생성일

### 1.4 MVP 제외 범위 (Out of Scope)
- 실시간 매칭 고도화(점수 모델), 다중 추천 알고리즘
- 외부 기관 신청 상태 자동 동기화
- 고급 알림 채널(SMS/카카오톡) 및 다국어 지원
- 정부/기관 데이터 완전 자동 수집 파이프라인(초기에는 수동/샘플 데이터 사용)

### 1.5 MVP 성공 지표
- 가입 완료율 40%+, 프로필 완성률 80%+
- 첫 매칭까지 평균 3분 이내
- 검색 사용률 60%+, 사용자당 북마크 3개+
- 캘린더 조회율 40%+
- 알림 오픈율 40%+, 재방문율 60%+

## 2) MVP 실행 계획

### 2.1 작업 방식 및 원칙
- **2주 스프린트** 기준으로 기능 단위 릴리스
- 프론트/백엔드 병행 개발, API 계약 우선
- 데이터는 **초기 샘플 데이터** 기반으로 시작하고 점진적 수집 확장

### 2.2 마일스톤 및 일정 (예시, 총 8주)

**M0 (주 1) - 준비/정의**
- 요구사항 확정, MVP 범위 동의
- UX 흐름/와이어프레임 작성
- API 스펙 초안(리소스/엔드포인트 정의)
- 샘플 혜택 데이터 스키마 확정

**M1 (주 2~3) - 기반 구축**
- Frontend 기본 레이아웃/라우팅 구성
- Backend 기본 프로젝트 세팅 및 사용자/혜택 API 골격
- 인증(JWT) 및 프로필 저장 기능
- 샘플 데이터 적재 스크립트

**M2 (주 4~5) - 탐색/상세/캘린더**
- 혜택 탐색(검색/필터/정렬) UI + API
- 혜택 상세 페이지 및 신청 링크 연결
- 캘린더 월별 뷰 및 마감 임박 강조

**M3 (주 6~7) - 내 혜택/알림**
- 북마크 및 상태 변경 기능
- 마감 알림 스케줄링(일 1회 배치, 이메일/앱 알림 1채널)
- 대시보드 통계 위젯 노출

**M4 (주 8) - 품질/릴리스**
- QA/버그 픽스, 성능 점검
- 기본 모니터링 지표 설정
- MVP 릴리스 및 사용자 피드백 수집

### 2.3 상세 실행 작업 (백로그)

**A. 제품/디자인**
- [ ] IA 기반 화면 구조 확정 및 정보 우선순위 정의
- [ ] 모바일 내비게이션(하단 탭/햄버거) 와이어프레임
- [ ] 핵심 컴포넌트(카드, 필터, 캘린더) 디자인 시스템 정의

**B. 프론트엔드**
- [ ] 공통 레이아웃(상단바/탭바/사이드바) 구성
- [ ] 혜택 탐색 페이지(검색, 필터, 정렬, 페이지네이션)
- [ ] 혜택 상세 페이지(상세 정보 탭, 제출서류 리스트)
- [ ] 캘린더 페이지(월간 그리드, 모달 리스트)
- [ ] 내 혜택 탭 및 상태 변경
- [ ] 알림 설정 UI

**C. 백엔드**
- [ ] 사용자/프로필 CRUD API
- [ ] 혜택 조회 API(검색/필터/정렬)
- [ ] 혜택 상세 API
- [ ] 북마크/상태 변경 API
- [ ] 알림 스케줄링(배치) 및 발송 기록

**D. 데이터/운영**
- [ ] 샘플 혜택 데이터 정의 및 적재
- [ ] 매칭 규칙(기본 규칙: 연령/지역/카테고리 필터)
- [ ] 모니터링 지표(탐색/북마크/캘린더 조회)

### 2.4 역할/책임 (예시)
- **PM**: 스코프 관리, 지표/리서치, 백로그 우선순위
- **디자인**: 와이어프레임/컴포넌트, UX 검증
- **프론트엔드**: UI 구현, API 연동, 접근성 대응
- **백엔드**: API/DB/배치, 보안/인증
- **QA**: 시나리오 테스트, 회귀 테스트

### 2.5 완료 정의 (Definition of Done)
- 핵심 사용자 여정 4가지가 오류 없이 동작
- 각 화면별 빈 상태/에러 상태 처리
- 기본 성능: 목록 조회 95p 500ms 이내
- 주요 이벤트 로깅 및 지표 수집 가능
- 배포 후 1주 내 피드백 수집 계획 수립

### 2.6 위험/대응
- **데이터 부족** → 초기에는 수동 데이터로 시작, 수집 파이프라인을 M3 이후 점진 적용
- **알림 채널 제약** → 이메일 1채널 우선 제공, SMS/카카오톡은 후속 확장
- **정책/요건 변경** → 필터/자격 요건을 JSON 형태로 유연하게 관리
